# Carpybara — Data Logic & Character Mapping Design

> ESP32 기반 OBD2 실시간 시각화 가젯의 데이터 처리 로직 및 캐릭터 엔진 구조 설계 문서

---

## 프로젝트 개요

**Carpybara**는 차량의 OBD2 포트에서 실시간 주행 데이터를 읽어, 각 데이터에 대응하는 캐릭터의 움직임과 애니메이션으로 시각화하는 ESP32 기반 소형 대시보드 가젯입니다.

macOS의 **RunCat** (CPU 사용량 → 고양이 달리기 속도)에서 영감을 받아, 이를 자동차 데이터 전반으로 확장한 멀티 캐릭터 시스템입니다.

```
OBD2 포트 → 데이터 수신 → 보정/스무딩 → 캐릭터 엔진 → 화면 출력
```

---

## 1. 데이터 모드 및 캐릭터 매핑

총 6개의 독립 모드를 지원하며, 각 모드는 고유한 OBD2 PID, 캐릭터, 시각화 로직을 가집니다. **캐릭터 에셋은 교체 가능한 구조**로 설계되어야 합니다.

### Mode 1 — Speed `PID: 010D`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 카피바라 |
| 단위 | km/h |
| 시각화 | 속도에 비례한 이동 속도 및 애니메이션 프레임 속도 변화 |

**상태 구간 예시**

| 속도 (km/h) | 상태 | 애니메이션 |
|------------|------|-----------|
| 0 (장시간) | SLEEP | 드러눔, ZZZ |
| 0 (단시간) | IDLE | 멍하니 대기 |
| 1 – 20 | WALK | 느릿느릿 걷기 |
| 21 – 60 | RUN | 달리기 |
| 61 – 100+ | TURBO | 전력질주, 볼살 날림 |
| 급감속 감지 | BRAKE | 앞발로 버티기 |

---

### Mode 2 — Coolant Temperature `PID: 0105`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 펭귄 |
| 단위 | °C |
| 시각화 | 온도 구간에 따른 **배경 테마 전환** |

**온도-테마 매핑**

| 온도 구간 | 배경 테마 | 설명 |
|----------|----------|------|
| < 60°C | 빙하 | 엔진 웜업 중 |
| 60 – 100°C | 정상 | 정상 운영 온도 |
| > 100°C | 용암 | 과열 경고 |

---

### Mode 3 — RPM `PID: 010C`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 시바견 |
| 단위 | RPM |
| 시각화 | 엔진 회전수에 따른 역동적 움직임 (꼬리 흔들기 속도, 귀 펄럭임 등) |

**RPM 매핑 예시**

| RPM 구간 | 동작 |
|---------|------|
| < 800 | 아이들링, 조용히 앉음 |
| 800 – 3000 | 꼬리 흔들기 |
| 3000 – 5000 | 빠른 꼬리 + 귀 펄럭 |
| > 5000 | 흥분 상태, 전신 흔들기 |

---

### Mode 4 — Throttle Position `PID: 0111`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 토끼 |
| 단위 | % (0 – 100) |
| 시각화 | 가속 페달 개도량에 따른 **즉각적** 점프/반응 |

> 다른 모드와 달리 스무딩을 최소화하여 드라이버 입력에 즉각 반응하는 것이 핵심입니다.

---

### Mode 5 — MAF (Mass Air Flow) `PID: 0110`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 용 |
| 단위 | g/s |
| 시각화 | 흡기 유량(엔진 부하)에 따른 **화염 이펙트 크기** 변화 |

**MAF-화력 매핑**

| MAF (g/s) | 화염 크기 |
|----------|----------|
| 낮음 | 불씨 수준 |
| 중간 | 안정적 화염 |
| 높음 | 대형 화염 폭발 이펙트 |

> MAF는 차량마다 출력 범위가 크게 다르므로 캘리브레이션 필수 (섹션 3 참조)

---

### Mode 6 — Engine Load `PID: 0104`

| 항목 | 내용 |
|------|------|
| 캐릭터 후보 | 쇠똥구리 |
| 단위 | % (0 – 100) |
| 시각화 | 엔진 부하량에 따른 **짐의 크기 또는 표정 변화** |

**부하-표현 매핑**

| 부하 (%) | 표현 |
|---------|------|
| 0 – 30 | 빈 손으로 여유롭게 걷기 |
| 31 – 70 | 적당한 짐을 굴리며 걷기 |
| 71 – 100 | 거대한 짐, 땀 흘리며 힘겨워하는 표정 |

## 2. 백그라운드 감시 시스템

### Control Module Voltage `PID: 0142`

이 데이터는 독립 캐릭터 모드로 노출되지 않지만, **시스템 전체를 상시 감시**합니다.

```
[메인 루프와 별도 태스크로 실행]

while (running):
    voltage = read_pid(0x0142)
    if voltage < CRITICAL_THRESHOLD:
        trigger_critical_state()
    sleep(VOLTAGE_CHECK_INTERVAL)
```

**임계값 및 대응 액션**

| 전압 (V) | 상태 | 트리거 |
|---------|------|--------|
| ≥ 12.0 | 정상 | 없음 |
| 11.5 – 12.0 | 경고 | 경고 아이콘 오버레이 |
| < 11.5 | 위험 | 현재 캐릭터 → 해골 상태로 강제 전환 + 경고창 |

> 위험 상태 해제는 전압이 12.0V 이상으로 **일정 시간 이상** 유지되어야 합니다 (히스테리시스 적용).

## 3. 데이터 보정 및 최적화

### 3-1. 캘리브레이션 (학습 모드)

MAF, Engine Load, Coolant Temp 등은 차량 기종마다 실제 출력 범위가 다릅니다. 주행 중 데이터를 수집해 해당 차량에 맞는 스케일을 자동 산출합니다.

```
// 캘리브레이션 데이터 구조 (각 PID별)
struct CalibrationData {
    float observed_min;
    float observed_max;
    float running_average;
    uint32_t sample_count;
};

// 정규화 공식
float normalized = (raw_value - observed_min) / (observed_max - observed_min);
```

**캘리브레이션 플로우**

```
초기 주행 시작
    │
    ▼
원시 데이터 수집 (min/max/avg 갱신)
    │
    ├─ 샘플 수 < 최소치 → 기본 범위값 사용
    │
    └─ 샘플 수 충분 → 관찰 범위로 정규화
         │
         ▼
      애니메이션 스케일 적용
```

- 캘리브레이션 데이터는 플래시(NVS/SPIFFS)에 차량별로 저장
- 재시동 시 이전 값 로드 → 누적 학습 구조

---

### 3-2. 스무딩 필터 (Weighted Moving Average)

OBD2 센서의 노이즈를 제거하고 애니메이션을 부드럽게 만들기 위해 가중 이동 평균을 적용합니다.

```
// 지수 가중 이동 평균 (EMA)
// alpha 값이 클수록 최신 데이터에 더 반응

smoothed = alpha * raw_value + (1 - alpha) * previous_smoothed
```

**모드별 alpha 권장값**

| 모드 | Alpha | 이유 |
|------|-------|------|
| Speed | 0.3 | 자연스러운 가속/감속 표현 |
| Coolant Temp | 0.1 | 온도 변화는 느리므로 강한 평활화 |
| RPM | 0.4 | 엔진 반응성 유지 |
| Throttle | 0.7 | 운전자 입력에 즉각 반응 |
| MAF | 0.4 | 엔진 부하 반응성 유지 |
| Engine Load | 0.3 | 안정적 표현 |

## 4. 캐릭터 엔진 구조 (Character Engine)

캐릭터 에셋을 쉽게 교체할 수 있도록 데이터 로직과 렌더링 레이어를 분리합니다.

```
┌─────────────────────────────────┐
│         OBD2 Data Layer         │  ← PID 요청/수신, 원시값 파싱
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│      Calibration & Smoothing    │  ← 보정 + EMA 필터
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│       State Mapping Engine      │  ← 정규화 값 → 캐릭터 상태 결정
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│       Character Asset Layer     │  ← 상태별 스프라이트/프레임 로드
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│         Display Renderer        │  ← TFT/OLED 화면 출력
└─────────────────────────────────┘
```

**캐릭터 교체를 위한 인터페이스 예시**

```c
typedef struct {
    const uint8_t* frames[];   // 상태별 스프라이트 배열
    uint8_t frame_count[];     // 상태별 프레임 수
    uint16_t frame_delay_ms;   // 기본 프레임 딜레이
} CharacterAsset;

// 새 캐릭터 등록 예시
CharacterAsset penguin_asset = {
    .frames = { penguin_idle, penguin_normal, penguin_overheat },
    .frame_count = { 2, 3, 4 },
    .frame_delay_ms = 100
};
```

## 5. 긴급 상태 (Critical State) 로직

전압 이상 외에도 다른 데이터에서도 위험 상태를 감지할 수 있도록 확장 가능한 구조로 설계합니다.

```
Critical State 진입 조건 (예시):
  - 전압 < 11.5V
  - 냉각수 온도 > 110°C
  - (향후 확장) DTC 오류 코드 감지

Critical State 동작:
  1. 현재 모드 렌더링 일시 중단
  2. 해골/경고 애니메이션 오버레이
  3. 경고음 (버저) 출력
  4. 복구 조건 충족 시 이전 모드 복귀
```

## 6. 향후 고려사항

- **멀티 PID 동시 수집**: 배경 감시(전압)와 활성 모드 PID를 비동기로 요청하여 폴링 주기 최적화
- **절전 모드**: SLEEP 상태 일정 시간 지속 시 화면 밝기 감소 또는 슬립
- **설정 UI**: 모드 전환, 캘리브레이션 초기화 등을 위한 버튼/터치 인터페이스
- **로그 저장**: SD카드 또는 BLE를 통한 주행 데이터 기록
