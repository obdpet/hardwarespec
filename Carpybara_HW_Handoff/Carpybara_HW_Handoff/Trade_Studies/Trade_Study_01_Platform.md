# Carpybara — Trade Study #1: 개발 플랫폼 선정

> **결정 사항**: Carpybara의 프로토타입 및 양산 플랫폼을 무엇으로 할 것인가?  
> **작성일**: 2026-04-14  
> **제약 조건**: PBL 데드라인 1~2개월 / 소매가 $50 / 사업화 병행

---

## 0. Trade Study란?

하드웨어 엔지니어가 부품이나 설계 방향을 결정할 때 사용하는 **표준 의사결정 방법론**입니다.

핵심 원리:
1. **감으로 고르지 않는다** — 평가 기준을 먼저 정하고, 점수로 비교한다
2. **가중치가 결론을 만든다** — 같은 부품이라도 프로젝트 상황에 따라 정답이 달라진다
3. **문서로 남긴다** — "왜 이걸 골랐는가"를 나중에 설명할 수 있어야 한다

> Trade Study는 "정답을 찾는 도구"가 아니라 **"오답을 걸러내는 도구"**입니다.
> 최종 판단은 여전히 사람이 하지만, 근거가 투명하게 남습니다.

---

## 1. 문제 정의 (Problem Statement)

Carpybara는 OBD2 데이터를 읽어 캐릭터 애니메이션을 출력하는 차량용 가젯이다.  
프로토타입 데드라인이 1~2개월이며, 동시에 양산 사업화도 고려해야 한다.  
교수님은 컴퓨팅 파워를 이유로 Raspberry Pi를 제안하셨다.

**결정해야 할 것**: 프로토타입을 어떤 플랫폼으로 만들고, 양산은 어떤 플랫폼으로 갈 것인가?

---

## 2. 필수 요구사항 (Pass/Fail 필터)

이 조건을 **하나라도 만족하지 못하면 즉시 탈락**합니다.  
점수를 매기기 전에 후보를 걸러내는 1차 필터입니다.

| # | 필수 요구사항 | 이유 |
|---|-------------|------|
| R1 | **Bluetooth Classic SPP 지원** | OBD2 ELM327 어댑터가 BT Classic SPP 사용. BLE만 지원하면 OBD2 통신 불가 |
| R2 | **SPI 디스플레이 구동 가능** | ILI9341 2.4" TFT는 4-wire SPI |
| R3 | **ADC 1채널 이상** | 조도센서(TEMT6000) 아날로그 입력 |
| R4 | **PWM 2채널 이상** | 백라이트 MOSFET 제어 + 피에조 버저 |
| R5 | **USB 5V 전원 동작** | 차량 USB 포트 또는 시거잭 어댑터 |

### 필수 요구사항 필터 결과

| 플랫폼 | R1 BT Classic | R2 SPI | R3 ADC | R4 PWM | R5 USB 5V | 결과 |
|--------|:---:|:---:|:---:|:---:|:---:|:---:|
| **ESP32-WROOM-32E** | O | O | O | O | O | **PASS** |
| **Raspberry Pi 4/5** | O (BlueZ) | O | 주의* | O | O | **PASS** |
| **Raspberry Pi Zero 2 W** | O (BlueZ) | O | 주의* | O | O | **PASS** |
| ESP32-S3 | **X** (BLE만) | O | O | O | O | **FAIL** |
| ESP32-C3 | **X** (BLE만) | O | O | O | O | **FAIL** |
| RP2040 (Pico) | **X** (BT 없음) | O | O | O | O | **FAIL** |

> *RPi는 자체 ADC가 없음. 외부 ADC(MCP3008 등) 또는 I2C 센서로 대체 필요. 불가능하진 않지만 추가 부품/작업 필요.

**ESP32-S3, ESP32-C3, RP2040은 Bluetooth Classic 미지원으로 탈락.**

## 3. 후보 전략 (Candidates)

Pass/Fail을 통과한 플랫폼으로 구성할 수 있는 **3가지 전략**:

---

### Strategy A: ESP32 처음부터 끝까지

```
[프로토타입]                    [양산]
ESP32 DevKit + 브레드보드  →    ESP32-WROOM-32E 커스텀 PCB
+ 2.4" ILI9341 TFT             (동일 코드 그대로)
+ ELM327 BT 어댑터
```

- **언어**: C/C++ (Arduino 프레임워크 또는 ESP-IDF)
- **핵심 라이브러리**: TFT_eSPI (디스플레이), ELMduino (OBD2)
- **장점**: 프로토 = 양산. 전환 비용 제로
- **단점**: 임베디드 개발은 디버깅이 어려움 (시리얼 모니터 의존)
- **프로토타입 BOM**: ~$15~20 (DevKit $8 + TFT $8 + 기타)

---

### Strategy B: RPi 프로토타입 → ESP32 양산

```
[프로토타입]                    [양산]
Raspberry Pi 4 + 모니터  →     ESP32-WROOM-32E 커스텀 PCB
+ Python/pygame                 (C/C++로 전면 재작성)
+ ELM327 BT 어댑터
```

- **언어**: Python (프로토) → C/C++ (양산)
- **핵심 라이브러리**: pygame (그래픽), python-OBD (OBD2)
- **장점**: 빠른 프로토타입, 높은 그래픽 자유도
- **단점**: 양산 전환 시 코드 전면 재작성 (Linux→bare metal, Python→C++)
- **프로토타입 BOM**: ~$50~70 (RPi4 $45 + 디스플레이 $15 + 기타)

---

### Strategy C: RPi 프로토타입 → RPi Compute Module 양산

```
[프로토타입]                    [양산]
Raspberry Pi 4 + 모니터  →     RPi CM4 + 커스텀 캐리어 보드
+ Python/pygame                 (동일 코드 ~90% 재사용)
+ ELM327 BT 어댑터
```

- **언어**: Python 전 과정
- **핵심 라이브러리**: pygame (그래픽), python-OBD (OBD2)
- **장점**: 코드 재사용율 최고, 그래픽 성능 최고
- **단점**: CM4 모듈만 $25~45 → 양산 BOM이 $50 소매가에 근접/초과
- **프로토타입 BOM**: ~$50~70

## 7. 민감도 분석 (Sensitivity Analysis)

> **민감도 분석이란**: "내가 가중치를 잘못 정했으면 어떻게 되는가?"를 확인하는 과정.
> 결론이 가중치 하나 바꿨을 때 뒤집힌다면, 그 결론은 신뢰하기 어렵습니다.

### 시나리오 1: "개발 속도가 더 중요하다면?" (C1 가중치 5→7)

| 전략 | 기존 총점 | 변경 후 총점 |
|------|---------|------------|
| A: ESP32 only | 101 | 101 + 3×2 = **107** |
| B: RPi→ESP32 | 88 | 88 + 5×2 = **98** |
| C: RPi→CM4 | 84 | 84 + 5×2 = **94** |

→ **순위 변동 없음. A 여전히 1위.** 개발 속도를 더 높여도 전환 비용 페널티가 B를 끌어내림.

### 시나리오 2: "양산 BOM이 덜 중요하다면?" (C2 가중치 4→1)

이 시나리오는 "당장은 PBL 과제만 신경 쓰고, 사업화는 나중에"인 경우.

| 전략 | 기존 총점 | 변경 후 총점 |
|------|---------|------------|
| A: ESP32 only | 101 | 101 - 5×3 = **86** |
| B: RPi→ESP32 | 88 | 88 - 5×3 = **73** |
| C: RPi→CM4 | 84 | 84 - 2×3 = **78** |

→ **순위 변동 없음. A 여전히 1위.** 다만 A와 C의 차이가 줄어듦.

### 시나리오 3: "전환 비용이 덜 중요하다면?" (C3 가중치 4→1)

이 시나리오는 "프로토타입만 빨리 만들면 되고, 양산 포팅은 나중에 해도 된다"인 경우.

| 전략 | 기존 총점 | 변경 후 총점 |
|------|---------|------------|
| A: ESP32 only | 101 | 101 - 5×3 = **86** |
| B: RPi→ESP32 | 88 | 88 - 1×3 = **85** |
| C: RPi→CM4 | 84 | 84 - 4×3 = **72** |

→ **A와 B가 거의 동점 (86 vs 85).** 전환 비용을 무시하면 B 전략이 A와 대등해짐.
→ 이 말은: **"당장 PBL 데모만 필요하고, 양산 포팅은 먼 미래"라면 B도 합리적**이라는 뜻.

## 8. 분석 및 논의

### A가 이기는 이유

Strategy A(ESP32 only)가 1위인 핵심 이유는 **"전환 비용 제로"** 때문입니다.

프로토타입에서 만든 코드가 그대로 양산에 들어간다는 것은:
- 추가 개발 기간 없음
- "프로토에서는 됐는데 양산에서 안 된다" 리스크 없음
- 데모에서 보여준 것이 곧 최종 제품

### B가 매력적인 경우

민감도 분석 시나리오 3에서 보듯이, **전환 비용을 무시할 수 있다면** B가 거의 동점입니다.

이런 상황이라면 B가 합리적:
- PBL 데모가 최우선이고, "양산은 1년 후 이야기"인 경우
- 비즈니스 플랜에서 "프로토: RPi / 양산: ESP32"로 제시하고, 양산 코드는 아직 안 짜도 되는 경우

하지만 **진지하게 사업화를 고려한다면**, 언젠가 반드시 ESP32 포팅 비용을 치러야 합니다.

### C가 약한 이유

RPi CM4 양산은 **$50 소매가 목표와 근본적으로 충돌**합니다.
CM4 모듈만 $25~45인 상태에서 디스플레이, PCB, 케이스, 조립비까지 더하면 BOM이 $50을 초과합니다.
소매가를 $80~100으로 올리면 가능하지만, 목표 시장(자동차 가젯)에서 경쟁력이 떨어집니다.

### 교수님의 RPi 제안에 대해

교수님이 RPi를 제안하신 맥락을 정확히 이해해야 합니다:

| 교수님의 가능한 의도 | 그에 맞는 전략 |
|-------------------|-------------|
| "프로토타입을 빨리 만들어라" | B가 맞음 (RPi로 빠르게 데모) |
| "컴퓨팅 파워가 충분해야 한다" | Carpybara의 실제 요구사항 확인 필요 (아래 참조) |
| "양산까지 RPi로 가라" | C가 맞지만 $50 목표와 충돌 |

**Carpybara에 정말 RPi급 컴퓨팅 파워가 필요한가?**

| 기능 | 연산 부하 | ESP32로 가능? |
|------|---------|:---:|
| OBD2 BT SPP 수신 | 매우 낮음 (10~100Hz 시리얼) | O |
| EMA 스무딩 필터 | 무시할 수준 (곱셈 1회/사이클) | O |
| 상태 머신 (6모드) | 무시할 수준 (if/else) | O |
| 스프라이트 애니메이션 240×320 @15fps | **중간** (partial update 필수) | O (TFT_eSPI 검증됨) |
| 파티클/블렌딩 등 고급 이펙트 | 높음 | X |

→ **현재 설계 기준으로 ESP32는 충분합니다.** RPi급 파워가 필요한 건 "실시간 3D 렌더링"이나 "카메라 영상 처리" 같은 경우.

## 9. 추가 발견: ESP32-WROOM 메모리 한계 (Rev B 추가)

Trade Study 초판 이후 ESP32-WROOM의 실제 메모리 한계가 확인되었습니다.

### 문제

- ESP32-WROOM SRAM: 520KB (DRAM 320KB + IRAM 200KB)
- 풀스크린 16-bit 프레임 버퍼 (240×320): **153KB — DRAM의 48%**
- TFT_eSPI 공식 문서: WROOM에서 16-bit 스프라이트 최대 **~200×200px**
- BT+WiFi 스택까지 구동하면 그래픽 가용 메모리 더 감소
- **Mode 2 (배경 전환: 빙하→정상→용암) 풀스크린 갱신 불가**

### 해결: ESP32-WROVER-E (PSRAM 8MB)

| | WROOM-32E | WROVER-E |
|--|--|--|
| SRAM | 520KB | 520KB (동일) |
| **PSRAM** | **없음** | **8MB** |
| 풀스크린 16-bit 스프라이트 | 불가 | **가능** |
| 핀 호환 | — | WROOM과 호환 |
| 코드 호환 | — | 100% 동일 코드 |
| 단가 (10K) | ~$3.50 | ~$4.50~5.50 |

### Strategy A+ 신설

```
프로토타입: ESP32-WROVER DevKit (PSRAM 8MB, 메모리 여유 확보)
    ↓ (개발 완료 후 실제 메모리 사용량 측정)
    ├─ PSRAM 불필요 → 양산: WROOM ($3.50)
    └─ PSRAM 필요 → 양산: WROVER ($4.50~5.50, 여전히 저렴)
```

교수님의 "오버스펙 프로토 → 다운그레이드" 원칙을 따르면서 코드 재작성 0%.

### 수정된 평가 매트릭스

| 기준 | 가중치 | A: WROOM only | **A+: WROVER→판단** | B: RPi→ESP32 | C: RPi→CM4 |
|------|:---:|:---:|:---:|:---:|:---:|
| C1. 개발 속도 | 5 | 3 (15) | **3 (15)** | 5 (25) | 5 (25) |
| C2. 양산 BOM | 4 | 5 (20) | **5 (20)** | 5 (20) | 2 (8) |
| C3. 전환 비용 | 4 | 5 (20) | **5 (20)** | 1 (4) | 4 (16) |
| C4. 그래픽 품질 | 3 | 3 (9) | **4 (12)** | 4 (12) | 5 (15) |
| C5. 커뮤니티 | 3 | 4 (12) | **4 (12)** | 4 (12) | 3 (9) |
| C6. 부팅 시간 | 2 | 5 (10) | **5 (10)** | 3 (6) | 2 (4) |
| C7. 전력/발열 | 2 | 5 (10) | **5 (10)** | 3 (6) | 2 (4) |
| C8. 크기 | 1 | 5 (5) | **5 (5)** | 3 (3) | 3 (3) |
| **총점** | **24** | **101/120** | **104/120** | **88/120** | **84/120** |
| **비율** | | 84.2% | **86.7%** | 73.3% | 70.0% |

A+는 A의 모든 장점을 유지하면서 그래픽 품질 점수만 올라감 (PSRAM으로 풀스크린 가능).

## 10. 최종 권고 (Recommendation)

### 1순위 권고: Strategy A+ (ESP32-WROVER 프로토 → WROOM 또는 WROVER 양산)

**이유**:
1. 총점 1위 (104/120, 86.7%)
2. PSRAM 8MB로 메모리 걱정 없이 개발 가능
3. 프로토→양산 코드 재작성 0% (WROVER↔WROOM 핀/코드 호환)
4. 양산 BOM 차이 $1~2 (사업성에 영향 없음)
5. 교수님의 "오버스펙 프로토 → 다운그레이드" 원칙 완벽 준수
6. 메모리 측정 데이터 기반으로 WROOM/WROVER 양산 결정 (데이터 드리븐)

**리스크**:
- C/C++ 디버깅이 Python보다 어려움 (바이브코딩으로 일부 완화)
- 복잡한 이펙트(파티클, 블렌딩)에는 한계 (현재 설계 범위 내 충분)

### 대안: Strategy B (RPi 프로토 → ESP32 양산)

다음 조건이 **모두** 해당될 때만 B를 선택:
- PBL 데모 품질이 최우선이고, 양산 포팅은 6개월 이상 후
- 교수님이 RPi 사용을 강하게 요구
- 애니메이션 품질이 ESP32로 불충분하다고 판단될 때

### Strategy C는 비권고

$50 소매가 목표와 근본적으로 충돌. 소매가를 $80~100으로 변경하지 않는 한 선택 불가.
