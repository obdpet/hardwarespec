# Carpybara — Trade Study #3: OBD2 통신 방식 선정

> **결정 사항**: Carpybara가 차량 OBD2 데이터를 어떤 방식으로 수신할 것인가?  
> **작성일**: 2026-04-14  
> **제약 조건**: 소비자 제품 UX / $50 소매가 / ESP32-WROVER 플랫폼

---

## 0. Trade Study란?

하드웨어 엔지니어가 부품이나 설계 방향을 결정할 때 사용하는 **표준 의사결정 방법론**입니다.

핵심 원리:
1. **감으로 고르지 않는다** — 평가 기준을 먼저 정하고, 점수로 비교한다
2. **가중치가 결론을 만든다** — 같은 부품이라도 프로젝트 상황에 따라 정답이 달라진다
3. **문서로 남긴다** — "왜 이걸 골랐는가"를 나중에 설명할 수 있어야 한다

> Trade Study는 "정답을 찾는 도구"가 아니라 **"오답을 걸러내는 도구"**입니다.
> 최종 판단은 여전히 사람이 하지만, 근거가 투명하게 남습니다.

## 1. 문제 정의 (Problem Statement)

Carpybara는 OBD2 데이터를 읽어 캐릭터 애니메이션 속도를 제어하는 차량용 가젯이다.  
현재 프로토타입은 사용자가 Veepeak VP11 ELM327 BT 어댑터를 별도 구매하여 무선으로 연결하는 방식이다.  
양산 시에도 이 방식을 유지할 것인가, 아니면 CAN 트랜시버를 내장하여 OBD2 포트에 직접 연결할 것인가?

**결정해야 할 것**: OBD2 데이터 수신 방식 — 무선(사용자 어댑터) vs 유선(내장 CAN) 중 어느 아키텍처를 양산에 채택할 것인가?

## 1.5. 배경 지식: OBD2 통신은 어떻게 작동하는가?

### ELM327 어댑터 방식 (현재 프로토타입)

```
[차량 ECU] ←CAN bus→ [ELM327 어댑터] ←BT SPP→ [ESP32]
                      (OBD2 포트에 꽂힘)          (별도 USB 전원)
```

1. ELM327 칩이 AT 명령어(`"010D\r"`)를 받아 CAN 프레임으로 변환
2. 차량 ECU가 응답
3. ELM327이 응답을 다시 ASCII 텍스트로 변환하여 BT로 전송
4. **병목**: AT 명령어 ↔ ASCII 변환 오버헤드 (PID당 50~500ms)

### CAN 직결 방식 (상용 HUD 표준)

```
[차량 ECU] ←CAN bus→ [CAN 트랜시버] ←3.3V 로직→ [ESP32 TWAI]
                      (PCB에 내장)                (OBD2에서 전원도 수급)
```

1. ESP32가 CAN 프레임을 직접 전송: `0x7DF: [02 01 0D 00 00 00 00 00]`
2. ECU가 CAN 프레임으로 직접 응답: `0x7E8: [03 41 0D XX 00 00 00 00]`
3. **ELM327의 ASCII 변환 단계가 없음** → 3~5배 빠름
4. ESP32의 내장 TWAI(CAN 컨트롤러)가 하드웨어 레벨에서 처리

### OBD2 커넥터 핀아웃 (SAE J1962)

```
Pin 4:  Chassis Ground     → ESP32 GND
Pin 5:  Signal Ground      → ESP32 GND
Pin 6:  CAN High           → SN65HVD230 CANH
Pin 14: CAN Low            → SN65HVD230 CANL
Pin 16: Battery 12V        → Buck 컨버터 → 3.3V → ESP32 VCC
```

> **중요**: CAN 데이터 라인(Pin 6/14)은 저전압 차동 신호(0~3.5V)입니다.  
> 12V 레벨 시프터가 필요 없습니다. CAN 트랜시버(SN65HVD230)가 3.3V로 직접 변환합니다.  
> 12V는 Pin 16 전원 라인에서만 — Buck 컨버터(MP2315, 입력 4.5~24V)로 처리합니다.

## 2. 필수 요구사항 (Pass/Fail 필터)

| # | 필수 요구사항 | 이유 |
|---|-------------|------|
| R1 | **OBD2 Mode 01 PID 읽기 가능** | Carpybara의 6개 PID(속도, 수온, RPM, 스로틀, MAF, 엔진부하) + 배경 모니터(전압) |
| R2 | **ESP32-WROVER 호환** | Trade Study #1에서 확정된 플랫폼 |
| R3 | **SPI 디스플레이(ILI9341 40MHz)와 하드웨어 충돌 없음** | Trade Study #2에서 확정된 디스플레이 |
| R4 | **Arduino/ESP-IDF 라이브러리 또는 문서 존재** | HW 경험 0 팀 — 레퍼런스 없이 프로토콜 직접 구현 불가 |

### 필수 요구사항 필터 결과

| 후보 | R1 PID 읽기 | R2 ESP32 호환 | R3 SPI 충돌 없음 | R4 라이브러리 | 결과 |
|------|:---:|:---:|:---:|:---:|:---:|
| **A: ELM327 BT Classic** | O | O | O (BT는 SPI 안 씀) | O (ELMduino) | **PASS** |
| **B: ESP32 TWAI + SN65HVD230** | O | O | O (TWAI는 전용 GPIO) | O (esp32_obd2, ESP-IDF TWAI) | **PASS** |
| **C: WiFi ELM327** | O | O | 주의* | O (ELMduino WiFi) | **조건부 PASS** |
| **D: MCP2515 + TJA1050 SPI** | O | O | 주의** | O (MCP_CAN) | **조건부 PASS** |

> *Option C: ESP32는 WiFi를 동시에 하나의 네트워크에만 연결 가능. ELM327 WiFi AP에 연결하면 OTA 업데이트 등 다른 WiFi 기능 사용 불가.  
> **Option D: MCP2515는 SPI 버스 사용. 디스플레이(VSPI)와 다른 버스(HSPI)로 분리 가능하나, WiFi+MCP2515 SPI 충돌이 [문서화된 이슈](https://github.com/espressif/arduino-esp32/issues/1670).

## 3. 후보 통신 방식 (Candidates)

---

### Option A: ELM327 Bluetooth Classic (현재 프로토타입)

```
[사용자가 할 일]
1. Veepeak VP11 어댑터 구매 (~$15)
2. 어댑터를 OBD2 포트에 꽂기
3. ESP32와 BT 페어링 (PIN: 1234)
4. Carpybara에 별도 USB 전원 연결
= 총 2개 기기 설치, BT 페어링 필요
```

- **통신 스택**: BluetoothSerial → ELMduino → AT 명령어
- **폴링 속도**: 저가 클론 3~5 PIDs/sec / 고급(OBDLink) 10~20 PIDs/sec
- **장점**: 프로토타입 코드 그대로. Carpybara BOM에 OBD2 비용 $0
- **단점**: 사용자가 호환 어댑터 별도 구매 필요. ELM327 클론 품질 편차 극심 (가품, 프리즈, BT 드롭). 고객 지원 부담 최대

**ELM327 클론 문제 상세**:
- 대부분의 "v2.1" 클론은 실제로 v1.0/v1.5 펌웨어 — 크랙된 PIC 마이크로컨트롤러
- 멀티 ECU 차량에서 프리즈
- BT 모듈(HC-05 클론) 진동 중 패킷 드롭
- ESP32 BluetoothSerial 연결 실패 빈발 (ESP_SPP_DISCOVERY_COMP_EVT 에서 멈춤)

---

### Option B: ESP32 TWAI + SN65HVD230 (CAN 직결)

```
[사용자가 할 일]
1. Carpybara에 포함된 OBD2 케이블을 차량 OBD2 포트에 꽂기
= 끝. 케이블 1개로 데이터 + 전원 동시 수급
```

- **통신 스택**: ESP32 TWAI 하드웨어 → CAN 프레임 직접 송수신
- **필요 부품**: SN65HVD230 CAN 트랜시버 ($0.29) + OBD2 커넥터/케이블 (~$1.50) + 120Ω 종단 저항
- **폴링 속도**: 15~30 PIDs/sec (ECU 응답시간 30~60ms가 병목, 하드웨어가 아님)
- **장점**: 상용 차량 HUD 표준 방식. 사용자 추가 구매 없음. 케이블 1개 = 데이터+전원. 폴링 3~5배 빠름. CS 부담 최소
- **단점**: OBD2 케이블 포함 필요 (BOM +~$2~3). CAN만 지원 (2008+ US 차량 ~95%, 2016+ 100%). 프로토타입 코드(ELMduino) 변경 필요

**CAN 트랜시버 후보**:

| IC | Vcc | ESP32 직결 | 10K 단가 | 비고 |
|---|---|:---:|---|---|
| **TJA1051T/3** (NXP) | 3.3~5V | O | ~$0.28 | **1st Source.** "/3" = 3.3V I/O. **TXD Dominant Timeout 내장** — ESP32 크래시 시 CAN 버스 블로킹 자동 방지. CAN-FD 지원 |
| **SN65HVD230** (TI) | 3.3V | O | ~$0.29 | 2nd Source. 3.3V 네이티브, ESD 보호. 단, TXD Dominant Timeout 없음 — 별도 외부 보호 회로 필요 |
| ~~TJA1050~~ | 5V only | X | — | 레벨 시프터 필요 — 탈락 |
| ~~MCP2551~~ | 5V only | X | — | 레벨 시프터 필요 — 탈락 |

**OBD2 PID 직접 요청/응답 예시 (CAN ISO 15765-4)**:

```
[요청] CAN ID 0x7DF: 02 01 0D 00 00 00 00 00
                      ↑  ↑  ↑
                      길이 Mode01 PID(속도)

[응답] CAN ID 0x7E8: 03 41 0D 3C 00 00 00 00
                      ↑  ↑  ↑  ↑
                      길이 응답  PID  값(60km/h)
```

### Option C: WiFi ELM327

```
[사용자가 할 일]
1. WiFi ELM327 어댑터 구매
2. 어댑터를 OBD2 포트에 꽂기
3. ESP32가 어댑터 WiFi AP에 연결 (IP: 192.168.0.10, Port: 35000)
4. Carpybara에 별도 USB 전원 연결
```

- **통신 스택**: WiFiClient → TCP → ELMduino AT 명령어
- **폴링 속도**: 3~10 PIDs/sec (FORScan 측정: 저가 클론 115ms+ 핑)
- **장점**: ELMduino 코드 재사용 (WiFiClient로 교체만). iOS 호환
- **단점**: WiFi 채널 충돌 — ESP32가 어댑터 AP에 연결하면 OTA 업데이트 등 다른 WiFi 불가. BT보다 더 불안정 (TCP keepalive 문제). 기술적 설정 장벽 높음

---

### Option D: MCP2515 SPI + TJA1050

```
[하드웨어 구성]
ESP32 → SPI → MCP2515(외부 CAN 컨트롤러) → TJA1050(트랜시버) → OBD2
```

- **통신 스택**: SPI → MCP_CAN 라이브러리 → CAN 프레임
- **필요 부품**: MCP2515 ($0.99) + TJA1050 ($0.15) + OBD2 커넥터/케이블 (~$1.50)
- **장점**: MCP_CAN 라이브러리 매우 성숙 (Arduino 생태계 표준)
- **단점**: **ESP32에는 TWAI가 내장되어 있어 MCP2515가 불필요.** SPI 버스 추가 점유. WiFi+MCP2515 SPI 충돌 문서화됨. RX 버퍼 2개뿐 — 고부하 시 프레임 드롭. BOM 비용이 Option B보다 높음

> **MCP2515는 TWAI가 없는 MCU(ATmega328 등)를 위한 부품입니다.**  
> ESP32에는 CAN 컨트롤러가 내장되어 있으므로, 동일 기능을 더 비싸고 복잡하게 구현하는 것에 불과합니다.

## 8. 분석 및 논의

### B가 이기는 이유: "소비자 제품"이기 때문

**이 Trade Study의 핵심 인사이트**: Carpybara는 개발 보드가 아니라 **소비자 제품**입니다.

ELM327 BT 방식은 개발자에게는 편합니다 — 라이브러리 있고, 코드 간단하고, BOM 추가 없고.  
하지만 **사용자**에게는 최악입니다:
- 호환 어댑터를 직접 알아서 구매해야 합니다
- BT 페어링을 직접 해야 합니다
- 연결이 안 되면 "어댑터 문제인지 제품 문제인지" 모릅니다
- Amazon 리뷰에 "doesn't work with my adapter"라고 씁니다

상용 차량 HUD 제품(ACECAR, Geloo, HUDWAY Drive $279)이 **전부** OBD2 직결 방식을 쓰는 이유가 있습니다. 소비자에게 서드파티 어댑터를 요구하는 제품은 시장에서 살아남지 못합니다.

### 프로토타입 전략: ELM327 BT → TWAI 점진적 전환

프로토타입 코드를 즉시 폐기할 필요는 없습니다:

```
1. EVT (현재): ELM327 BT + ELMduino (프로토타입 코드 유지)
2. DVT: SN65HVD230 브레이크아웃 보드 구매 → TWAI 코드 개발 + 테스트
3. PVT: 양산 PCB에 SN65HVD230 내장 + OBD2 케이블 커넥터
4. ELMduino 코드는 개발/디버그 모드로 보존 (ifdef로 전환 가능)
```

코드 변경 범위:
- **변경**: OBD2 수신 레이어 (ELMduino AT 명령어 → TWAI CAN 프레임)
- **재사용 100%**: EMA 스무딩, 상태 머신, 캐릭터 엔진, 디스플레이 렌더러
- **예상 소요**: 2~3일 (esp32_obd2 라이브러리 활용 시)

### Trade Study #1 R1(BT Classic SPP)에 대한 영향

Trade Study #1에서 **Bluetooth Classic SPP**를 필수 요구사항(R1)으로 설정하고, 이를 근거로 ESP32-S3, ESP32-C3, RP2040을 탈락시켰습니다.

양산에서 TWAI를 채택하면 BT Classic은 더 이상 OBD2 통신에 필요하지 않습니다. 이론적으로는 ESP32-S3(BLE only, TWAI 있음)도 양산 후보가 될 수 있습니다.

**하지만 MCU 변경은 권고하지 않습니다:**
- ESP32-WROVER는 이미 Trade Study #1에서 확정
- WROVER에 BT Classic이 있어도 해가 되지 않음 (안 쓰면 그만)
- MCU를 바꾸면 Trade Study #1부터 다시 해야 함
- 프로토타입(ELM327 BT) 동작을 위해 BT Classic은 여전히 유용

### Option D (MCP2515)를 선택하지 않는 이유

Option D는 B와 동일한 "유선 직결" UX를 제공하지만, **ESP32에 이미 TWAI가 있으므로 불필요한 IC(MCP2515)를 추가하는 것**에 불과합니다.

| | B (TWAI) | D (MCP2515) |
|--|--|--|
| CAN 컨트롤러 | ESP32 내장 | 외부 IC ($0.99) |
| SPI 버스 사용 | 안 함 (전용 GPIO) | 사용 (디스플레이 SPI와 경쟁) |
| BOM 추가 | ~$2~3 | ~$3~4 |
| WiFi 충돌 | 없음 | 있음 (문서화) |
| RX 버퍼 | 하드웨어 큐 | 2개 (프레임 드롭 리스크) |

**B가 모든 면에서 D보다 낫거나 같습니다.** D를 선택할 합리적 이유가 없습니다.

### 전원 아키텍처 변경 (후속 검토 필요)

TWAI 방식 채택 시 전원 구조가 변경됩니다:

| | 현재 (ELM327 BT) | TWAI 직결 |
|--|--|--|
| 일상 전원 | USB 5V (시거잭/USB 포트) | **OBD2 Pin 16 (12V)** |
| 펌웨어 플래싱 | USB-C (CH340C) | USB-C (CH340C, 동일) |
| Buck 입력 | 5V | **12V** (MP2315 입력 4.5~24V — 커버됨) |
| 사용자 케이블 수 | 2개 (USB + 어댑터) | **1개** (OBD2 케이블) |

MP2315의 입력 범위(4.5~24V)가 12V를 커버하므로 Buck 컨버터 교체는 불필요합니다.  
단, 피드백 저항 값 변경(출력 전압 설정)과 입력 캡 전압 등급 확인이 필요합니다.  
**이 변경은 별도 문서(전원 설계 검토)에서 다룹니다.**

## 9.5. CAN 버스 안전성 (Rev B 추가)

Carpybara가 차량 CAN 버스에 직접 연결하므로, **차량 통신에 간섭하거나 노이즈를 유발할 리스크**를 반드시 분석하고 대응해야 합니다.

### 리스크 1: Bus Dominant Fault (가장 심각)

**문제**: ESP32 펌웨어가 크래시하면서 TWAI TX 핀이 dominant(Low) 상태로 고정되면, CAN 버스 전체가 블로킹됨. 차량 ECU 간 통신(ABS, 에어백, 엔진 제어)이 방해받을 수 있음.

**이것이 SN65HVD230 → TJA1051T/3 교체의 핵심 이유입니다.**

| | SN65HVD230 (기존 1st) | TJA1051T/3 (변경 후 1st) |
|--|--|--|
| TXD Dominant Timeout | **없음** | **있음 (tdom)** |
| ESP32 크래시 시 동작 | TX dominant 유지 → **버스 블로킹** | tdom 경과 후 자동 recessive → **버스 보호** |
| 자동차 안전 등급 | 산업용 | **AEC-Q100 자동차 등급** |

**추가 SW 보호**: 펌웨어에서도 TWAI 드라이버에 watchdog timer를 설정하여, TX 고정 상태를 감지하면 TWAI 페리페럴을 리셋.

### 리스크 2: Bus Loading (버스 과부하)

**문제**: 과도한 OBD2 PID 폴링이 CAN 버스 대역을 점유하여 ECU 간 정상 통신을 지연시킬 수 있음.

**정량 분석**:
- CAN 버스 대역: 500 kbps (ISO 15765-4)
- OBD2 요청 1건 (요청+응답): ~200 bit
- 초당 10회 폴링: 2,000 bit/sec = **버스 대역의 0.4%** → 영향 없음
- 초당 30회 폴링: 6,000 bit/sec = **1.2%** → 여전히 안전
- ECU 간 일반 통신이 버스의 30~50%를 점유하므로, 1~2%는 무시 가능

**대응**: 
- 펌웨어에서 **폴링 레이트를 10Hz로 하드 제한**
- 7개 PID를 순차 폴링하면 PID당 ~1.4Hz → 애니메이션 반응성에 충분
- ISO 15765-4에서 정의한 P2 타이밍(ECU 응답 대기 최대 50ms)을 준수

### 리스크 3: EMI / 전기적 노이즈

**문제**: OBD2 포트에서 송풍구까지 ~1.5m 케이블이 안테나처럼 작동하여 노이즈를 수신하거나 방사할 수 있음. 차량 내부는 점화 시스템, 모터, 릴레이 등 EMI 소스가 많음.

**대응**:
1. **차폐 트위스트 페어 케이블** — CANH/CANL을 꼬아서 외부 노이즈 상쇄. 차폐(shield)를 GND에 연결
2. **TVS 다이오드** — CAN 라인에 NXP PESD1CAN 배치. 차량 전기적 과도현상(load dump, 정전기) 흡수
3. **120Ω 종단 저항** — 점퍼/솔더 브릿지로 on/off 가능. OBD2 포트 위치에 따라 종단이 불필요할 수 있음
4. **커먼 모드 초크** — 선택 사항. 심한 EMI 환경에서 추가 보호. DVT 테스트 결과에 따라 결정

### CAN 안전 회로 종합

```
OBD2 Pin 6 (CANH) ──┬── [PESD1CAN TVS] ──┬── TJA1051T/3 CANH
                     │                     │
                   [120Ω]                   │
                   (점퍼)                    │
                     │                     │
OBD2 Pin 14 (CANL) ──┴── [PESD1CAN TVS] ──┴── TJA1051T/3 CANL
                                                    │
                                           TXD ←── ESP32 TWAI_TX
                                           RXD ──→ ESP32 TWAI_RX
                                           VCC ←── 3.3V
                                           GND ←── GND
```

## 10. 최종 권고 (Recommendation)

### 1순위 권고: Option B — ESP32 TWAI + TJA1051T/3 (CAN 직결, Bus Dominant 보호)

**양산 OBD2 통신 방식으로 확정합니다.**

**이유**:
1. 총점 1위 (119/130, 91.5%) — 2위(D)와 15점 차, 현 프로토타입(A)과 48점 차
2. 3개 민감도 시나리오 전부에서 1위 유지
3. **상용 차량 HUD 제품의 사실상 표준 방식** — ACECAR, Geloo, HUDWAY 등 전부 이 아키텍처
4. 사용자 경험 최고 — 케이블 1개로 데이터+전원 동시
5. 고객 지원 부담 최소 — "어댑터 호환성" CS 원천 제거
6. BOM 추가 ~$2~3 — $50 소매가에 영향 미미
7. 폴링 속도 3~5배 향상 — 캐릭터 애니메이션 반응성 직결

**필요 부품 (AVL 추가)**:
- **CAN 트랜시버**: TJA1051T/3 (1st, TXD Dominant Timeout 내장) / SN65HVD230DR (2nd)
- **OBD2 케이블**: J1962 Type A Male → 기기측 커넥터, ~1.5m
- **종단 저항**: 120Ω (점퍼로 on/off 전환 가능)

**프로토타입 전환 전략**:
1. EVT: ELM327 BT 유지 (현재 코드)
2. DVT: TJA1051T/3 또는 SN65HVD230 브레이크아웃 보드로 TWAI 코드 개발 (2~3일)
3. PVT: 양산 PCB에 TJA1051T/3 내장 + PESD1CAN TVS 다이오드

### Option A는 프로토타입 전용으로 유지

ELM327 BT + ELMduino 코드는 개발/디버그 모드로 보존합니다.  
양산 펌웨어에서 `#ifdef USE_ELM327` 으로 전환 가능하게 유지하면,  
개발 중 OBD2 케이블 없이도 ELM327으로 빠르게 테스트할 수 있습니다.

### Option C, D는 비권고

- **C (WiFi ELM327)**: A보다 모든 면에서 열위. WiFi 채널 충돌. 선택할 이유 없음
- **D (MCP2515)**: ESP32에 TWAI가 내장되어 있으므로 불필요한 IC 추가. B 대비 비용↑ 성능↓ 신뢰성↓
